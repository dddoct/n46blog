## 双模式设计

### 模式1：总检索

**设计目标**：全量备份所有成员的博客和图片

**API端点**：`https://www.nogizaka46.com/s/n46/api/list/blog`

**参数**：
- `page`: 页码（从1开始）

**特点**：
- 返回所有成员的博客，按时间倒序
- 每页100篇博客
- 适合全量爬取和增量更新

### 模式2：指定成员检索

**设计目标**：针对特定成员的个人博客页面进行定向爬取

**API端点**：`https://www.nogizaka46.com/s/n46/api/list/blog`

**参数**：
- `ct`: 成员ID（如55397）
- `page`: 页码（从1开始）

**特点**：
- 只返回指定成员的博客
- 使用成员个人页面API
- 适合追踪特定成员

**成员ID获取**：
- 方式1：从数据库查找（如果模式1已爬取过）
- 方式2：从API实时查找（调用通用API匹配成员名）

## 数据流说明

### 模式1：总检索数据流

```
API请求 (list/blog?page=N)
    │
    ▼
解析JSONP响应
    │
    ▼
提取博客数据 ──► 保存到数据库 (blogs表)
    │                ├─► 新博客：INSERT OR REPLACE
    │                └─► 已存在：UPDATE (保留原有ID)
    ▼
提取图片URL ────► 保存到数据库 (images表)
                     ├─► 新图片：INSERT (download_status = 0)
                     └─► 已存在：跳过 (UNIQUE约束: blog_id + original_url)
```

### 模式2：指定成员检索数据流

```
获取成员ID (数据库或API)
    │
    ▼
API请求 (list/blog?ct=成员ID&page=N)
    │
    ▼
解析JSONP响应
    │
    ▼
提取博客数据 ──► 保存到数据库 (blogs表)
    │                └─► INSERT OR REPLACE
    ▼
提取图片URL ────► 保存到数据库 (images表)
                     └─► INSERT (跳过已存在)
```

### 图片下载数据流

```
查询待下载图片 (download_status = 0)
    │
    ▼
按博客ID分组，计算图片序号
    │
    ▼
异步下载图片 ──► 保存到对应目录
    │              ├─► 模式1: output1/images/by_*/
    │              └─► 模式2: output2/images/成员名/日期/
    ▼
更新数据库状态 (download_status = 1, local_path = 路径)
```

## 数据库设计

### blogs 表（博客信息）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PRIMARY KEY | 博客唯一ID |
| title | TEXT | 博客标题 |
| author | TEXT | 作者名（日文） |
| author_id | TEXT | 作者ID（用于模式2） |
| publish_date | TEXT | 发布日期 |
| url | TEXT | 博客URL |
| content | TEXT | 博客内容（HTML） |
| crawled_at | TIMESTAMP | 爬取时间 |

### images 表（图片信息）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY AUTOINCREMENT | 自增ID |
| blog_id | TEXT | 所属博客ID |
| original_url | TEXT | 原始URL |
| local_path | TEXT | 本地存储路径 |
| download_status | INTEGER | 0=待下载, 1=已下载 |
| created_at | TIMESTAMP | 创建时间 |

**唯一约束**：`(blog_id, original_url)` - 防止同一图片重复记录

## 覆盖逻辑详解

### 博客数据覆盖逻辑

```sql
INSERT OR REPLACE INTO blogs (id, title, author, author_id, publish_date, url, content, crawled_at)
VALUES (?, ?, ?, ?, ?, ?, ?, CURRENT_TIMESTAMP)
```

**逻辑说明**：
- 新博客：插入新记录
- 已存在博客：更新所有字段（包括content），保留原有ID
- 用途：支持博客内容更新（如编辑后重新爬取）

### 图片数据覆盖逻辑

```sql
INSERT OR IGNORE INTO images (blog_id, original_url, download_status, created_at)
VALUES (?, ?, 0, CURRENT_TIMESTAMP)
```

**逻辑说明**：
- 新图片：插入新记录，状态为待下载(0)
- 已存在图片：跳过（不更新）
- 用途：防止重复下载同一图片

### 下载状态管理

| 状态 | 值 | 说明 |
|------|-----|------|
| 待下载 | 0 | 新发现的图片，尚未下载 |
| 已下载 | 1 | 已成功下载到本地 |

**重新下载**：
- 如需重新下载，可重置状态：`UPDATE images SET download_status = 0`
- 或删除数据库重新爬取

## 成员ID获取机制

### _get_member_id() 方法逻辑

```python
def _get_member_id(self, member_name):
    # 方式1：从数据库查找
    SELECT DISTINCT author_id FROM blogs 
    WHERE author LIKE '%成员名%' LIMIT 1
    
    # 方式2：从API实时查找
    调用 list/blog?page=1
    遍历返回数据，匹配 member_name in item.get('name', '')
    返回 item.get('arti_code')
```

### 成员ID来源

| 来源 | 说明 | 优先级 |
|------|------|--------|
| 数据库 | 模式1爬取时存储的author_id | 优先 |
| API实时 | 调用通用API匹配成员名 | 备选 |

## 使用场景示例

### 场景1：首次全量备份

```bash

python main.py full --pages 20

# 查看统计
python main.py stats
```

### 场景2：日常增量更新

```bash
# 爬取最新5页
python main.py crawl --pages 5

# 下载新图片
python main.py download
```

### 场景3：追踪特定成员

```bash
# 获取池田瑛紗最近3页博客
```

```bash
python main.py export
```

## 技术要点

### JSONP解析

API返回格式：`res({"count": 100, "data": [...]});`

解析步骤：
1. 去除前缀 `res(`
2. 去除后缀 `);`
3. JSON解析剩余内容

### 异步下载

使用 `aiohttp` 实现并发下载：
- 默认并发数：5
- 失败重试：3次
- 请求延迟：1秒

### 编码处理

Windows终端GBK编码问题处理：
- 过滤emoji字符
- 使用 `encode('gbk', errors='ignore')`

## 注意事项

1. **成员ID缓存**：模式2首次使用某成员时，如果数据库中没有该成员ID，会从API实时获取
2. **页数控制**：模式2默认3页，可通过 `--pages` 调整
3. **存储隔离**：两种模式输出到不同目录（output1/ 和 output2/），互不干扰
4. **数据库共享**：两种模式共用同一个数据库（output1/data/n46_blog.db）
